# 计费与兑换开发文档

## 开发目标

基于 `计费与兑换需求文档.md`，实现一套可上线的站内余额、兑换码、模型动态扣费和轻量管理员后台。第一版保持数据结构适中，不拆过细，但必须保证兑换码校验和扣费逻辑不能被前端绕过。

## 阶段一：Supabase 数据结构升级

目标：补齐计费、兑换和管理员配置所需数据表。

开发内容：

- 扩展 `user_accounts`，增加 `role` 字段，默认 `user`。
- 新增 `credit_packages` 表，保存点数套餐。
- 新增 `redeem_codes` 表，保存兑换码、点数、金额、状态、使用人和使用时间。
- 新增 `model_pricing` 表，保存模型成本、倍率和参数匹配规则。
- 新增 `site_settings` 表，保存客服二维码、客服微信号和充值说明。
- 配置 RLS 策略：
  - 普通用户只能读写自己的账户。
  - 管理员可管理套餐、兑换码、模型价格和站点配置。

验收标准：

- Supabase SQL 可一次性执行成功。
- 普通用户不能读取其他用户账户。
- 管理员账号可访问后台配置数据。

## 阶段二：管理员权限与后台入口

目标：实现仅管理员可访问的后台页面。

开发内容：

- 在前端读取当前用户 `user_accounts.role`。
- 当 `role = admin` 时显示管理员入口。
- 新增管理员后台视图，例如 `admin` section 或独立页面。
- 非管理员访问后台时显示无权限提示。
- 支持退出后台返回普通工作台。

验收标准：

- 普通用户看不到管理员入口。
- 管理员可以进入后台。
- 未登录用户不能访问后台。

## 阶段三：客服配置与点数套餐管理

目标：让客服二维码和售卖套餐可动态配置。

开发内容：

- 后台支持配置客服微信号、二维码图片 URL、充值说明文案。
- 前台点数充值页面从 Supabase 读取客服配置。
- 后台支持新增、编辑、启用、停用点数套餐。
- 前台展示启用中的套餐，例如：

```text
9.9 元 = 990 点
39 元 = 3900 点
99 元 = 9900 点
```

验收标准：

- 修改客服二维码后，前台刷新即可生效。
- 停用的套餐不在前台展示。
- 点数套餐金额和点数展示准确。

## 阶段四：兑换码生成与服务端兑换

目标：替换当前前端 Mock 兑换码逻辑，改为 Supabase 持久化校验。

开发内容：

- 后台按套餐生成单个或批量兑换码。
- 兑换码包含：`code`、`package_id`、`credits`、`price_cny`、`status`、`used_by`、`used_at`。
- 新增服务端兑换接口或 Supabase RPC，例如 `redeem_credit_code`。
- 兑换逻辑必须原子执行：
  - 校验兑换码存在。
  - 校验状态为 `unused`。
  - 标记兑换码为 `used`。
  - 增加用户点数。
  - 写入账户 `ledger`。
- 前台移除 `mockRedeemCodes`。

验收标准：

- 有效兑换码可增加点数。
- 同一个兑换码只能兑换一次。
- 已使用兑换码不能被其他用户重复兑换。
- 浏览器前端不能通过改代码直接增加点数。

## 阶段五：模型价格配置与预计扣费

目标：根据模型实际成本和利润倍率动态计算扣点。

开发内容：

- 后台维护 `model_pricing`：
  - 模型名称
  - 类型：图片或视频
  - 清晰度
  - 时长
  - 比例
  - 成本金额
  - 利润倍率
  - 是否启用
- 前台生图、生视频表单根据当前参数匹配价格配置。
- 按公式计算扣点：

```text
扣点 = ceil(成本金额 × 利润倍率 × 100)
```

- 生成按钮附近展示预计消耗：

```text
预计消耗：400 点（约 4.00 元）
```

验收标准：

- 不同模型和参数展示不同预计消耗。
- 后台修改价格后，前台刷新生效。
- 未启用或未配置价格的模型不能提交生成。

## 阶段六：生成扣费与失败退款

目标：生成任务和用户余额建立真实扣费关系。

开发内容：

- 提交生成前判断余额是否充足。
- 余额不足时阻止生成，并提示去充值。
- 余额充足时先扣除预计点数并写入流水。
- 生成任务提交成功后保存历史项目。
- 任务提交失败时立即退回点数。
- 任务最终状态为失败时退回点数并更新项目状态。
- 点数流水增加类型：
  - `redeem`
  - `generate`
  - `refund`

验收标准：

- 余额不足无法生成。
- 生成成功会扣除对应点数。
- 任务失败会自动退回点数。
- 刷新或重新登录后余额、流水、历史项目仍然正确。

## 阶段七：后台查询与运营校验

目标：方便管理员核对兑换码、用户余额和模型成本。

开发内容：

- 后台展示兑换码列表，支持按状态筛选。
- 展示兑换码使用人和使用时间。
- 展示点数套餐列表。
- 展示模型价格列表。
- 展示客服配置预览。
- 可选：展示最近点数流水和用户余额。

验收标准：

- 管理员能查到兑换码是否已使用。
- 管理员能确认某个兑换码发放的点数和价格。
- 管理员能核对当前模型扣费配置。

## 阶段八：上线验收与安全检查

目标：保证计费和兑换逻辑上线前安全可控。

开发内容：

- 检查所有 Supabase RLS 策略。
- 确认前端没有保留 Mock 兑换码。
- 确认扣点和兑换逻辑不依赖前端可信输入。
- 补充 `.env.example` 中 Supabase 和 APIMart 配置说明。
- 执行：

```bash
pnpm lint
pnpm exec tsc --noEmit
pnpm build
```

验收标准：

- 普通用户不能访问管理员数据。
- 普通用户不能篡改余额。
- 兑换码不能重复使用。
- 生成扣费、失败退款、历史持久化均通过测试。
- Vercel 环境变量配置完整后可正常部署。

## 测试重点

- 注册登录后账户数据是否隔离。
- 兑换码是否只能使用一次。
- 点数套餐金额是否和到账点数一致。
- 生成前预计消耗是否准确。
- 余额不足是否阻止生成。
- APIMart 任务失败是否退款。
- 刷新页面和重新登录后，余额、流水和历史项目是否仍然存在。
- 管理员后台是否只有 admin 用户可见。
