# 计费与兑换需求文档

## 背景与目标

当前产品通过 APIMart 调用图片和视频模型，不同模型、清晰度、时长会产生不同成本。为了避免高成本模型被低价调用导致亏损，系统需要建立一套按实际成本动态扣费的点数体系，并支持客服人工收款后发放兑换码。

目标是实现：用户购买点数、使用兑换码充值、生成内容时按模型成本扣点、失败自动退回点数，并支持管理员配置套餐、兑换码、客服信息和模型成本。

## 点数与金额关系

点数本质上作为站内余额的最小单位。

统一规则：

```text
1 点 = 0.01 元
100 点 = 1 元
```

用户购买套餐时，金额直接换算为点数。例如：

```text
9.9 元 = 990 点
39 元 = 3900 点
99 元 = 9900 点
```

前台可以同时展示点数和约等金额，例如：

```text
当前余额：3900 点（约 39.00 元）
预计消耗：400 点（约 4.00 元）
```

## 模型扣费规则

系统不固定“生图一次多少点”或“视频一次多少点”，而是按模型实际成本和利润倍率动态计算。

扣点公式：

```text
扣点 = ceil(模型实际成本金额 × 利润倍率 × 100)
```

示例：

```text
APIMart 实际成本：2.00 元
利润倍率：2.0
用户扣费金额：4.00 元
实际扣点：400 点
```

每个模型参数组合都应有独立配置，例如模型、类型、清晰度、时长、比例、成本金额、利润倍率、是否启用。

## 生成扣费流程

用户点击生成前，系统根据当前选择的模型和参数计算预计消耗。

流程：

1. 查询模型价格配置。
2. 计算预计扣点。
3. 判断用户余额是否充足。
4. 余额不足时阻止生成，并提示充值。
5. 余额充足时先扣除点数并写入流水。
6. 提交 APIMart 生成任务。
7. 任务成功后保留扣费。
8. 任务提交失败或最终失败时，自动退回点数并写入退款流水。

第一版可以采用“提交前直接扣点，失败后自动退回”的方式。

## 兑换码流程

当前人工客服收款模式下，用户不会直接在线支付，而是联系微信客服付款后获得兑换码。

用户流程：

1. 用户在点数充值页面查看客服二维码。
2. 用户联系微信客服并完成付款。
3. 客服根据用户购买套餐发放兑换码。
4. 用户在站内输入兑换码。
5. 系统校验兑换码有效且未使用。
6. 兑换成功后增加对应点数，并记录点数流水。

兑换码必须由服务端或 Supabase RPC 校验，不能只在前端判断，避免用户篡改前端逻辑刷点数。

## 管理员后台需求

需要一个最小管理员后台，用于维护人工收款相关配置。

第一版后台功能：

1. 客服配置
   - 上传或填写客服二维码图片 URL。
   - 配置客服微信号。
   - 配置充值说明文案。

2. 点数套餐管理
   - 配置套餐名称。
   - 配置售价金额。
   - 配置到账点数。
   - 配置是否启用。

3. 兑换码管理
   - 按套餐生成单个或批量兑换码。
   - 查看兑换码状态：未使用、已使用、已禁用。
   - 查看使用用户和使用时间。

4. 模型价格配置
   - 配置模型名称。
   - 配置内容类型：图片或视频。
   - 配置清晰度、时长、比例等参数。
   - 配置实际成本金额。
   - 配置利润倍率。
   - 配置是否启用。

## 数据存储设计

现有用户数据保存在 Supabase。

已有表：

```text
user_accounts
```

保存内容：

```text
user_id
credit_balance
projects
ledger
redeemed_codes
```

后续新增表建议：

```text
redeem_codes
credit_packages
model_pricing
site_settings
```

其中：

- `redeem_codes` 保存兑换码、点数、金额、状态、使用人和使用时间。
- `credit_packages` 保存人工售卖套餐。
- `model_pricing` 保存模型成本、倍率和扣费规则。
- `site_settings` 保存客服二维码和站点配置。

管理员权限可以先通过 `user_accounts.role` 实现，取值为：

```text
user
admin
```

管理员账号可以先在 Supabase Table Editor 中手动设置为 `admin`。

## 验收标准

1. 用户注册登录后，点数、历史项目、兑换记录持久化到 Supabase。
2. 用户刷新页面或重新登录后，余额和历史项目仍然存在。
3. 兑换码只能使用一次。
4. 已使用兑换码不能被其他用户再次兑换。
5. 生成前展示预计消耗点数和约等金额。
6. 余额不足时不能提交生成任务。
7. 任务失败时自动退回已扣点数。
8. 管理员可以配置客服二维码、点数套餐、兑换码和模型价格。
9. 前端不能直接决定兑换码是否有效，也不能直接绕过扣费逻辑。
